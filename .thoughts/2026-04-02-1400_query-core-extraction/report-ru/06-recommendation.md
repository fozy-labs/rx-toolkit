# Рекомендация

## Основной вариант: Подход D — Утилитарные функции (Utility Functions) (немедленно)

Извлечь две самостоятельные функции: `cleanupLifecycleResolvers()` и `createLifecycleTools()`. Это устраняет ~19 из 57 буквально идентичных строк — наиболее плотные, явно дублирующиеся блоки — с помощью ~15 строк нового кода и **нулевых структурных изменений**.

**Почему это правильное решение:**

- Подтверждённое дублирование составляет 57 идентичных + 6 структурно схожих строк из 545 суммарных строк (~10,5%). Подходы A–C вводят иерархии классов, связанность через защищённые поля или инфраструктуру композиции, которые сравнимы по объёму с дублированием или превышают его.
- Утилитарные функции — это паттерн, используемый TanStack Query (самостоятельные вспомогательные функции, без общей базы наблюдателя) и RTK Query (общая функция `handleNewKey` для `onCacheEntryAdded`). Ни одна библиотека из сравнительной матрицы не решает эту задачу через промежуточный класс.
- Нулевой риск: чистые функции, независимо тестируемые, без изменений роли `CacheEntry` как обобщённого реактивного контейнера.
- Обратная совместимость: гарантирована. Никаких изменений публичного API, сигнатур типов, иерархии.

**Оценка трудозатрат:** Минимальная — один файл, две функции, обновление двух точек вызова, добавление юнит-тестов.

## Вторичный вариант: Повторная оценка после завершения Фазы 2

Все четыре класса машин состояний Command содержат комментарии «Phase 2 stub». Они функционально завершены для семантики одноразовых мутаций (one-shot mutation), но структурно расходятся с машинами Resource (нет `MachineWithData`, нет состояния обновления, нет SSR). Извлечение общей инфраструктуры из нестабильной цели означает повторное извлечение, когда заглушки разовьются.

**Когда возвращаться к вопросу:**
- После реализации Фазы 2 Command и стабилизации структуры машин состояний.
- Если планируется конкретный третий тип сущности (InfiniteQuery, Subscription), Подход A (обогащение `CacheEntry` полями abort + resolver) становится оправданным.
- Если после Фазы 2 машины Command приобретут значительное сходство с Resource, пересмотреть Подход B — но только тогда.

## Чего НЕ следует делать

| Антипаттерн | Причина |
|---|---|
| Промежуточный класс `FetchableCacheEntry` (B) | Трёхуровневая иерархия для 2 потребителей ~57 общих строк. Вектор ошибок: `_abortInflight()` обнуляет контроллер, нарушая проверку устаревания по идентичности в Resource. Заявлено сокращение на 65 строк — невозможно при 57 фактически идентичных строках без протаскивания недублированного кода в базовый класс. |
| Композиция `FetchEngine` (C) | Шаблонный код связывания заменяет дублирование 1:1. Суммарное количество строк возрастает. `resolveDataLoaded(data)` не проще, чем `this._entryDataLoaded.resolve(data)` — косвенность без упрощения. |
| Унификация машин состояний | Ни одна библиотека этого не делает. Асимметрия намеренная: запросы (queries) имеют устаревание/обновление/SSR; мутации — одноразовые. |

## Матрица принятия решений

| Если... | Тогда... |
|---|---|
| Цель — только DRY | Подход D сейчас — постоянное решение |
| Планируется третий тип сущности | Подход D сейчас, Подход A после Фазы 2 |
| Машины Command развиваются с высоким сходством с Resource | Подход D сейчас, пересмотр B после Фазы 2 |
| Третий тип сущности не планируется никогда | Подход D — постоянное решение |

## Урок RTK Query

RTK Query доказывает, что более глубокое совместное использование (sharing) кода ВОЗМОЖНО — единственный `executeEndpoint` обрабатывает запросы, мутации и бесконечные запросы с ~90% общего кода. Его обработчики `onQueryStarted` и `onCacheEntryAdded` полностью разделяются во время выполнения с расхождением только на уровне типов. API жизненного цикла rx-toolkit был смоделирован по образцу RTK Query.

Однако RTK Query использует Redux как инфраструктуру — единый конвейер middleware, диспатчащий через один store. rx-toolkit использует сигналы и потоки RxJS на каждый экземпляр. Архитектурная основа принципиально различается. **Следует перенять паттерн вспомогательных функций жизненного цикла RTK** (общий `handleNewKey`, общая настройка промисов), **а не его подход монолитного middleware**.

## Открытые вопросы для владельца проекта

1. **Планируется ли третий тип сущности?** (InfiniteQuery, Subscription, StreamResource) — это единственный сценарий, при котором структурные затраты Подхода A оправданы.
2. **Должна ли Фаза 2 Command привести структуру машин состояний ближе к Resource?** (базовый класс `MachineWithData`, метод `cloneWith`, состояние обновления) — определяет, будет ли расти поверхность для будущего извлечения.
3. **Что является основной целью — DRY или расширяемость?** Если DRY: Подхода D достаточно. Если расширяемость: дождаться Фазы 2, чтобы выявить стабильные общие паттерны.
