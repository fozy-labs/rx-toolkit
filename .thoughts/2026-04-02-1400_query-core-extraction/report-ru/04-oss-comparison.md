## Матрица сравнения

| Аспект | TanStack Query v5 | RTK Query | Apollo Client | SWR v2 | urql | rx-toolkit |
|---|---|---|---|---|---|---|
| **Совместно используемая база (shared base)** | `Subscribable` + `Removable` (~65 строк) | `CommonEndpointDefinition` + единый `executeEndpoint` (~90% совместно используемого рантайма) | Монолитный `QueryManager` (~1850 строк, без разделения) | `_internal`-ядро: `Cache`, `internalMutate`, `serialize` | `Operation` с тегом `kind`, единый `executeRequestOperation` | Базовый `CacheEntry` (~72 строки), раздельные иерархии |
| **Машины состояний (machines)** | Встроенные редьюсеры на каждую сущность, **не разделяются** | Отдельные Redux-слайсы, общий enum `QueryStatus` | `ObservableQuery` (~1700 строк) vs встроенный Promise (~180 строк) | Нет машины — плоские объекты `{ data, error }` | Нет машины — `OperationResult` — плоские данные | Иммутабельные иерархии классов (4+4), **не разделяются** |
| **Кэш (cache)** | Два отдельных: `QueryCache` (Map) + `MutationCache` (Set) | Единый редьюсер, отдельные под-слайсы (`queries` + `mutations` + `provided`) | Единый `InMemoryCache`, общие оптимистичные слои | Единая плоская `Map<string, State>`, общее пространство ключей | По одному на exchange; `cacheExchange` хранит запросы, инвалидирует при мутации | Отдельный `CacheMap` на каждый Resource; `Map` на каждый Command |
| **Совместное использование жизненного цикла (lifecycle sharing)** | Раздельное: авто-запрос + таймеры устаревания vs бездействие-до-мутации | **Общее**: `onQueryStarted` + `onCacheEntryAdded` для обоих, единый конвейер middleware | Раздельное: `ObservableQuery` vs одноразовый Promise; общий `broadcastQueries()` | Раздельные хуки; `internalMutate` — общий путь записи | Единый `Client` dispatch, расходящиеся dedup + teardown | Раздельное, но параллельное: оба имеют `onQueryStarted` + `onCacheEntryAdded` (~57 строк дублирования) |
| **Плагины / расширения (plugin / extension)** | Нет системы плагинов; адаптеры фреймворков оборачивают наблюдателей | `buildCreateApi(...modules)` — компонуемая модульная система | Цепочка `ApolloLink` + `typePolicies` | Массив `use: [...middlewares]` | Конвейер exchange (функции `ExchangeIO`) | `IPlugin` с `augmentResource` + `augmentCommand` |
| **React-привязка (React binding)** | `useSyncExternalStore`, оборачивающий наблюдателей ядра | `useSelector` + сгенерированные хуки для каждого endpoint | `ObservableQuery` → обёртка `useQuery` | `useSyncExternalStore` + отслеживание зависимостей | Потоки Wonka → хуки | Сигналы + `useSyncExternalStore` |

## Профили библиотек

**TanStack Query v5.** Два крошечных базовых класса (`Subscribable` для pub/sub, `Removable` для сборки мусора) — всё остальное независимо: классы сущностей, наблюдатели, кэши, редьюсеры, типы результатов. Наиболее близка к текущей структуре rx-toolkit, где query и mutation совместно используют лишь минимальный реактивный контейнер. Ключевой вывод: *намеренное дублирование* между `QueryObserver` (745 строк) и `MutationObserver` (227 строк) — общего базового наблюдателя нет, несмотря на структурное сходство. **Достоверность: Высокая.**

**RTK Query.** Наиболее глубокое совместное использование в экосистеме: единый payload creator `executeEndpoint` обрабатывает thunk-и как для query, так и для mutation (~90% общего кода); callback-и жизненного цикла `onQueryStarted` и `onCacheEntryAdded` проходят через единый обработчик middleware, который сопоставляет оба типа thunk через `isPending(queryThunk, mutationThunk)`. Разделение происходит на уровне слайсов (отдельные редьюсеры, селекторы, хуки) и в стратегии ключей кэша (хеш аргументов vs requestId). API жизненного цикла rx-toolkit было спроектировано по образцу RTK Query, что делает его наиболее архитектурно релевантным ориентиром. **Достоверность: Высокая.**

**Apollo Client.** Нет разделения query/mutation на уровне ядра — монолитный `QueryManager` (~1850 строк) обрабатывает всё. Реальная асимметрия — на уровне результатов: `ObservableQuery` (богатый, долгоживущий, ~1700 строк) vs встроенный Promise мутации (fire-and-forget, ~180 строк). Единый `InMemoryCache` с общими оптимистичными слоями. Ключевой вывод: монолитный подход работает, но делает tree-shaking невозможным, а код мутаций занимает ~10% от инфраструктуры запросов по весу. **Достоверность: Высокая.**

**SWR v2.** Мутация — буквально middleware, оборачивающее хук запроса (`useSWRMutation = withMiddleware(useSWR, mutation)`). Общее: плоский key-value `Cache`, `internalMutate` для записи, `serialize` для ключей. Однако мутация поддерживает собственное состояние через `useStateWithDeps`, отдельно от подписки на кэш запросов. Ключевой вывод: общий путь записи, раздельное поведение чтения. **Достоверность: Высокая.**

**urql.** Потоковая унификация через exchange-и: каждый запрос становится `Operation` с явным тегом `kind` (`query`/`mutation`/`subscription`), отправляемым через `executeRequestOperation`. Exchange-и фильтруют по `operation.kind` — то есть операции совместно используют конвейер, но немедленно расходятся. Ключевой вывод: диспетчеризация по тегу kind — чистый подход, но фильтрация на каждом exchange воссоздаёт разделение. **Достоверность: Высокая.**

## Две референсные модели

### 1. TanStack Query — минимальная общая база

Совместно используется лишь ~65 строк инфраструктуры. Иерархии сущностей, наблюдатели, кэши и машины состояний полностью независимы. Это подтверждает текущий подход rx-toolkit с `CacheEntry` (~72 строки общего кода) — масштаб совместного использования сопоставим. TanStack доказывает, что раздельное хранение query/mutation работает в масштабе (20k+ звёзд на GitHub, адаптировано для 4 фреймворковых адаптеров). Стоимость — структурное дублирование (~50% логики наблюдателей параллельна), принятое как компромисс ради простоты.

### 2. RTK Query — глубокое совместное использование жизненного цикла

Совместно используется ~90% рантайма исполнения через `executeEndpoint`, плюс унифицированный middleware жизненного цикла для `onQueryStarted`/`onCacheEntryAdded`. Это напрямую релевантно, поскольку rx-toolkit скопировал дизайн API жизненного цикла RTK Query. RTK Query доказывает, что более глубокое совместное использование инфраструктуры жизненного цикла жизнеспособно и хорошо проверено в продакшене. Граница разделения проходит на уровне редьюсеров, селекторов и хуков — не на уровне исполнения или жизненного цикла.

## Консенсус экосистемы (уточнённый)

Экосистема НЕ сходится к единому ответу. Два жизнеспособных паттерна сосуществуют:

1. **Минимальная база** (TanStack, urql, SWR): совместно используется лишь тончайшая инфраструктура (pub/sub, сборка мусора, примитивы кэша). Машины, логика запросов и жизненный цикл остаются раздельными. Оптимизирует простоту и независимость.

2. **Общее исполнение + жизненный цикл** (RTK Query): совместно используется исполнитель запросов и механизм хуков жизненного цикла; разделение — на уровне состояния/селекторов/хуков. Оптимизирует согласованность и уменьшает дублирование жизненного цикла.

**Универсальное согласие**: ни одна библиотека не использует совместно машины состояний для запросов и мутаций. Enum-ы статусов могут совпадать, но логика редьюсеров всегда расходится (авто-запрос + отслеживание устаревания vs fire-and-forget + состояние простоя).

## Уроки для rx-toolkit

| Источник | Урок | Применимость |
|---|---|---|
| TanStack | ~65 строк общей базы достаточно; не стоит чрезмерно абстрагировать наблюдателей/машины | Подтверждает текущий размер `CacheEntry`; подтверждает, что машины должны оставаться раздельными |
| RTK Query | Middleware жизненного цикла можно унифицировать для обоих типов сущностей без слияния логики состояний | Напрямую применимо — rx-toolkit уже имеет параллельные `onQueryStarted`/`onCacheEntryAdded`; общие хелперы жизненного цикла доказали жизнеспособность |
| RTK Query | `executeEndpoint` ветвится по типу сущности в рантайме — работает при ~90% общего кода | Рассмотреть для общей оркестрации запросов, если дублирование вырастет за пределы текущих ~57 строк |
| Apollo | Монолитный подход создаёт технический долг в tree-shaking и поддерживаемости | Подтверждает: избегать слияния Resource + Command в единый класс |
| SWR | Чистые утилитарные функции (`internalMutate`, `serialize`) — наиболее легковесный механизм совместного использования | Подтверждает подход утилитарных функций для очистки жизненного цикла (~15 строк, нулевые структурные изменения) |
| urql | Модель `Operation` с тегом kind — чистый подход, но exchange-и немедленно фильтруют — разделение по-прежнему реально | Не стоит ожидать, что унифицированная абстракция «операции» устранит логику, специфичную для каждого типа |
