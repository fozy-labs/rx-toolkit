# Cross-check: 01-architecture × 04-decisions

## Inconsistencies Found

| # | Location (file:section) | Location (file:section) | Description |
|---|------------------------|------------------------|-------------|
| 1 | 01-architecture:§2 diagram ("Layer 5: plugins/"), §2 table (plugins depends on "core/, react/") | 04-decisions:ADR-1 ("Plugins alongside api/") | **Plugin layer position conflict.** Architecture places plugins as Layer 5 — above react/ (Layer 4) — and states plugins depend on `core/` and `react/`. ADR-1 places plugins "alongside api/" (Layer 3). If plugins are at the api level, they cannot depend on the react layer (Layer 4) per the "only depend on layers below" rule. Architecture's own dependency arrows confirm `RHP → useResAgent` (plugins → react) which contradicts ADR-1's placement. |
| 2 | 01-architecture:§2 text ("strict 4-layer architecture") | 04-decisions:ADR-1 Decision ("4-layer strict hierarchy") | **Layer count vs diagram.** Both files state "4-layer", but architecture §2 diagram explicitly labels 5 layers (lib=1, core=2, api=3, react=4, plugins=5). The table enumerates 6 rows (adding types/). ADR-1 defines 4 layers and treats plugins as an addendum "alongside api", not as a numbered layer. The diagram contradicts both the architecture text and ADR-1. |
| 3 | 01-architecture:§5.2 (`_lastEntry$: SignalFn<ResourceV2CacheEntry \| null>`), §6.1 (same) | 04-decisions:ADR-11 (`_lastEntry$: SignalFn<CacheEntry \| null>`) | **_lastEntry$ generic type mismatch.** Architecture types `_lastEntry$` as `SignalFn<ResourceV2CacheEntry<TArgs,TData> \| null>` (subclass). ADR-11 types it as `SignalFn<CacheEntry \| null>` (base class). These are structurally different — `CacheEntry` is broader than `ResourceV2CacheEntry`. |
| 4 | 01-architecture:§5.2 CacheEntry class (`-_signal$: SignalFn<TState>`) | 04-decisions:ADR-5 code sample (`this._state$`, `this._value$`) | **CacheEntry private field naming mismatch.** Architecture's class diagram names the internal signal field `_signal$`. ADR-5's implementation code uses `_state$` (the source BehaviorSubject piped through share) and `_value$` (the resulting observable). Neither name matches `_signal$`. |
| 5 | 01-architecture:§5.1 MachineWithData (`+patchState: TPatchState<TData> \| null`, `+createPatch()`), §7.2 ("Patcher — internal to MachineWithData") | 04-decisions:ADR-6 ("ResourceV2CacheEntry stores `_patchState: TPatchState<TData> \| null`") | **patchState dual ownership.** Architecture §5.1 places `patchState` and `createPatch()` on MachineWithData (public). Architecture §5.2 simultaneously places `_patchState` and `createPatch()` on ResourceV2CacheEntry (private). ADR-6 states RCE stores _patchState and checks the consistency violation flag. Architecture §7.2 says "Patcher — internal to MachineWithData". Both classes own patchState and createPatch, creating an unclear ownership boundary. Who is authoritative for patch state — the immutable machine or the mutable RCE? |

Architecture §4 shows CacheEntry depending only on `signals/Signal` — no RxJS dependency is depicted. The Patcher→immer dependency IS shown, but CacheEntry→RxJS is absent. |
| 7 | 01-architecture:§5.2 (`_lastArgs: TArgs \| SKIP_TOKEN \| null`) | 04-decisions:ADR-10, ADR-15 (use `SKIP` consistently) | **SKIP vs SKIP_TOKEN naming.** Architecture §5.2 class diagram uses `SKIP_TOKEN` in the `ResourceV2Agent._lastArgs` type. Architecture's own public API (§7.1), lib layer (§2), and React integration (§6.3) all use `SKIP`. ADR-15 naming table lists `SKIP` as the sentinel. `SKIP_TOKEN` appears only in this one class diagram field and is inconsistent with both files' naming convention. |

## Verdict: INCONSISTENT

Seven inconsistencies found: two structural (plugin layer position/count), one type mismatch (_lastEntry$), one naming mismatch (CacheEntry fields), one ownership ambiguity (patchState), one missing dependency (RxJS), and one naming inconsistency (SKIP_TOKEN). Issues #1, #3, and #5 are high-severity as they affect architectural correctness and type safety. Issues #4 and #6 affect implementability (field names and dependency tracking). Issues #2 and #7 are medium-severity naming/diagram inconsistencies.
