## Issues

### 1. MachineWithData.createPatch return type mismatch

- **Architecture** (§5.1 class diagram): `+createPatch(patchFn): IPatchHandle | null`
- **Tests** (SM31): expects `success.createPatch(...)` → `{ machine, patchHandle }` — both a **new machine instance** and a handle
- **Problem**: Architecture return type omits the new machine instance. Since machines are immutable (transitions return new instances), `createPatch` must return the new machine alongside the handle. The class diagram is incomplete.
- **Severity**: **Critical** — affects Patcher/machine contract; implementer will not know the correct return shape from architecture alone.

### 2. ResourceV2 class diagram missing tested methods

- **Architecture** (§5.2 ResourceV2 class): public methods listed are `createAgent()`, `query()`, `getEntry()`, `getEntry$()`, `invalidate()`.
- **Tests** reference additional methods:
  - `resetCache()` — RE14, RE22, RE23
  - `cacheEntries()` — RE15
  - `hydrateEntry(args, machine)` — RE16
  - `hasEntry(args)` — RE18
- **Architecture** §7.2 mentions these as "internal" but doesn't list them anywhere (not in class diagrams, not in boundary definitions).
- **Severity**: **High** — four internal methods are tested but invisible in architecture; implementer has no specification for their signatures.

### 3. `createCacheMap()` factory not in architecture

- **Tests** (CM-F04, CM-F05): reference `createCacheMap({ keyStrategy: "serialize" | "compare" })` factory function.
- **Architecture**: shows `SerializeCacheMap` and `CompareCacheMap` classes directly in diagrams (§3, §5.2) but never introduces a `createCacheMap()` factory function.
- **Severity**: **High** — tests assume a factory abstraction that architecture doesn't define.

### 4. `IResourceV2CacheEntry` interface not in class diagrams

- **Tests** (AG14, RE07): reference `IResourceV2CacheEntry` as a consumer-facing interface type.
- **Architecture** (§5.2): shows `ResourceV2CacheEntry` class only, no corresponding `IResourceV2CacheEntry` interface. §9.4 mentions it in the legacy-comparison narrative but never defines it as part of the new design.
- **Severity**: **High** — public interface vs. implementation class distinction is missing from architecture.

### 5. Two hydration mechanisms — architecture shows only one

- **Architecture** (§7.1): lists `hydrateSnapshot()` as public API.
- **Tests**: describe **two** hydration paths:
  1. `hydrateSnapshot(api, snapshot)` — imperative, standalone function (SN03)
  2. `createApi({ initialSnapshot })` — lazy, per-resource consumption (AP08, AP08a, AP08b, AP08c, SN07, SN09, SN11, SN12)
- **Architecture** §3a `createApi(options)` diagram doesn't show `initialSnapshot` in options or `_savedSnapshot` internal field. The lazy per-resource consumption mechanism is prevalent in tests but absent from architecture.
- **Severity**: **High** — the dominant hydration path in tests (`initialSnapshot`) is not modeled in architecture.

### 6. Plugin tests mislabeled as "core Layer"

- **Architecture** (§2): Plugins are **Layer 5** (`plugins/`), explicitly separate from core (Layer 2).
- **Tests**: section header is "Test Cases — core Layer: Plugins" (PL01–PL08).
- **Severity**: **Medium** — layer mismatch in test organization vs. architecture layering. Plugins depend on react/ and core/ but belong to their own layer.

### 7. Snapshot tests not labeled with layer

- **Severity**: **Medium** — external dependency on RxJS operators is documented narratively (§9.3) but absent from the formal dependency diagram (§4).

### 13. `maxSnapshotDataAge` not in architecture

### 14. LifecycleHooks methods not in architecture class diagrams

- **Tests** (LH01–LH09): reference specific methods: `fireCacheEntryAdded(args, getCacheEntry)`, `fireQueryStarted(args, getCacheEntry)`, `clearAll()`, and promise-based hooks `$cacheDataLoaded`, `$cacheEntryRemoved`, `$queryFulfilled`.
- **Architecture** (§3, §5.2): `LifecycleHooks` appears only as a box in component diagrams (`LH["LifecycleHooks"]`). No class diagram defining its methods/interface exists. §6.2 shows `PromiseResolver` used by LifecycleHooks but doesn't specify the method signatures.
- **Severity**: **High** — LifecycleHooks is a fully-tested component with 9 test cases but has no architectural method specification.

### 15. `resetCache()` visibility ambiguity

- **Architecture** (§5.2): ResourceV2 class diagram only shows public methods — `resetCache()` is absent.
- **Architecture** (§7.1, §7.2): `resetCache()` is not listed as either public or internal API.
- **Tests** (RE14, RE22, RE23): test `resetCache()` directly on ResourceV2 instances.
- **API tests** (AP05): `api.resetAll()` delegates to `resetCache()` per resource.
- **Severity**: **Low** — the method is needed for `api.resetAll()` delegation but its visibility (public/internal) is undefined in architecture.
