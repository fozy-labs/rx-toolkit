## Issues Found

### Issue #1 ŌĆö UC-5: patchState location incorrect in comment
**Severity**: Medium
**Where**: `05-usecases.md` UC-5, edge case code comment (line ~291)
**What's wrong**: Comment says `// isConsistencyViolation is tracked in TPatchState on the machine state`. ADR-6 explicitly places `_patchState: TPatchState<TData> | null` as a private field of `ResourceV2CacheEntry`, not on the machine state. The machine tracks `TMachineState` (status, args, data, updatedAt); patchState is a separate concern owned by the cache entry.
**Expected**: `// isConsistencyViolation is tracked in TPatchState on the cache entry (ResourceV2CacheEntry._patchState)`
**Impact**: Could mislead implementer into putting patchState on the machine classes, violating the separation defined in ADR-6.

### Issue #2 ŌĆö UC-4: broken cross-reference to ADR-2
**Severity**: Low
**Where**: `05-usecases.md` UC-4, ref annotation after imperative retry example (line ~225)
**What's wrong**: `[ref: 04-decisions.md ADR-2 ŌĆö Refreshing errorHappened returns MachineSuccess with stale data]` ŌĆö ADR-2 does NOT describe "Refreshing errorHappened" transition behavior. ADR-2 covers the immutable class-based machine decision and TError removal. The "Refreshing ŌåÆ error ŌåÆ fallback to MachineSuccess with stale data" is a machine transition detail not present in ADR-2's text.
**Expected**: Either (a) reference the dataflow or model doc that describes this transition, or (b) add this transition behavior to ADR-2's Consequences section.

### Issue #3 ŌĆö Shared setup: missing `as const` for plugins array
**Severity**: Low
**Where**: `05-usecases.md` Shared Setup section (line ~53)
**What's wrong**: `plugins: [new ReactHooksPlugin()]` lacks `as const`. ADR-9 shows `[new ReactHooksPlugin()] as const` in its type-level mechanism example, and UC-10/UC-14 within the same file correctly use `as const`. Without `as const`, TypeScript widens the type and `PluginAugmentations` won't resolve plugin contributions to their specific types.
**Expected**: `plugins: [new ReactHooksPlugin()] as const` ŌĆö consistent with ADR-9 pattern and UC-10/UC-14 examples in the same document.

### Issue #4 ŌĆö UC-5: `getEntry(true)` invalid call for void-args resource
**Severity**: Medium
**Where**: `05-usecases.md` UC-5, edge case code (line ~284)
**What's wrong**: `const entry = todosResource.getEntry(true);` passes a boolean where `TArgs = void`. Comment says "void args: only doInitiate param needed" ŌĆö implying `getEntry` accepts `(doInitiate: boolean)` overload. However, UC-6 uses `getEntry({ id: "1" })` with args as the first parameter. The `getEntry` signature is `getEntry(args: TArgs): ResourceV2CacheEntry | null` ŌĆö for void args this should be `getEntry()` (no argument) or `getEntry(undefined)`, not `getEntry(true)`.
**Expected**: `const entry = todosResource.getEntry();` for void-args resource, or clarify in the model/API that `getEntry` has distinct overloads for different TArgs.
**Impact**: Type-unsafe example that wouldn't compile; confuses the `getEntry` contract.
