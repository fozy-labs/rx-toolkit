## Summary

**Overall consistency: MEDIUM-HIGH.** Core types, states, and public interfaces are well-covered by test cases. Three direct contradictions found; the majority of warnings relate to machine-level method signatures that exist in tests but have no corresponding model interfaces.

### Critical Issues (FAIL)

1. **#28 — SM24 property name**: `refreshing.patches` references non-existent top-level property. Model has `patchState: TPatchState<TData> | null` with `patches` nested inside. Fix test assertion.
2. **#29 — AG12 `state$.obs`**: References `.obs` property not defined on `ComputeFn` in the model. Either document the property in the model or fix the test to use the correct subscription API.
3. **#30 — LH06 resolution type**: "resolves with data" vs model's `Promise<{ data: TData }>`. Test description should specify the wrapper object.

### Structural Gap (WARN — recurring pattern)

Items #31–#34 share a common root cause: **the model defines machine *state types* but not machine *class method interfaces***. All transition methods (`start`, `successHappened`, `errorHappened`, `reset`, `invalidate`, `retry`) and data-manipulation methods (`createPatch → {machine, patchHandle}`, `finishPatch(type, patch)`, `abortAllPendingPatches()`) are defined only in the test cases. Consider adding a `IMachineIdle`, `IMachinePending`, etc. interface section to the model, or a transition table mapping methods to each machine class — this would anchor the test contracts in the model.
