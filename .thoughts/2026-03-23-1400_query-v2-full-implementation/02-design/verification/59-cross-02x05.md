## Inconsistencies

### #1 — resetAll() sequence vs GC cleanup: "abort patches"

**Severity**: Medium
**Dataflow**: §1.7 GC cleanup decisions explicitly states:
> **Abort patches — NO**: When refcount reaches 0 and the GC timer fires, no subscribers exist. `IPatchHandle` references are obtained through the entry and held by consumers (React components). If all consumers have unmounted (refcount=0), no one holds active patch handles — pending patches are impossible. No abort needed.

This reasoning applies to `CacheEntry.complete()` — the function that fires when the GC timer expires.

**Usecases**: UC-13 and UC-12 Step 3 both describe resetAll() as:
```
All cache entries: CacheEntry.complete() → abort patches → idle → fire onClean$ → delete
```

**Conflict**: Both paths invoke `CacheEntry.complete()`, but:
- Dataflow says complete() does NOT abort patches (because no subscribers hold patch handles at refcount=0)
- UC-13/UC-12 say complete() DOES abort patches (during resetAll, components may still be mounted with active patch handles)

The reasoning diverges because GC fires only at refcount=0 (no consumers), whereas resetAll fires while consumers may still exist. However, both reference the same function (`CacheEntry.complete()`). If `complete()` has a single implementation, it either always or never aborts patches.

**Resolution options**:
- (a) resetAll uses a different internal mechanism than GC's `complete()` — UC-13/UC-12 should not say `CacheEntry.complete()` for this step, or should describe a separate `reset()` + `complete()` sequence.
- (b) `complete()` always aborts patches (safe no-op when none exist) — dataflow §1.7 "abort patches — NO" should say "not needed" rather than "not done", clarifying that the code path exists but is a no-op during GC.

### #2 — Agent state `entry` field absent from dataflow outputs

**Severity**: Low
**Usecases**: UC-5 destructures `entry` from the hook result:
```typescript
const { entry } = useResourceV2Agent(todosResource);
// entry: IResourceV2CacheEntry<void, TodoList> | null
```

**Dataflow**: All `state$` outputs in §1.1–§1.6 show only `{status, data, isLoading, isInitialLoading, isRefreshing}`. No dataflow diagram includes `entry` as part of the agent state. Example from §1.1:
```
Agent-->>App: state$: {status:"success", data: TData, isLoading:false}
```

**Impact**: Dataflow diagrams do not represent the full shape of `IResourceV2AgentState`. A reader following only the dataflow would not know `entry` is accessible from the hook. Not a contradiction (dataflow shows a subset for brevity), but a gap.

### #3 — Inconsistent agent state field selection across dataflow diagrams

**Severity**: Low
**Dataflow** shows different subsets of agent state fields in different diagrams:

| Section | State shown | Fields included | Fields omitted |
|---------|------------|-----------------|----------------|
| §1.1 (initial fetch) | pending | `status, data, isInitialLoading` | `isLoading` |
| §1.1 (initial fetch) | success | `status, data, isLoading` | `isInitialLoading, isRefreshing` |
| §1.2 (SWR) | pending | `status, data, isLoading, isInitialLoading` | `isRefreshing` |
| §1.4 (invalidation) | refreshing | `status, data, isRefreshing` | `isLoading` |
| §1.6 (retry) | pending | `status, data, isLoading` | `isInitialLoading` |

**Usecases**: UC-11 provides a complete, consistent state table:

| Phase | `isLoading` | `isInitialLoading` | `isRefreshing` |
|-------|-------------|---------------------|----------------|
| Pending (initial) | `true` | `true` | `false` |
| Pending (SWR) | `true` | `false` | `false` |
| Refreshing | `true` | `false` | `true` |

**Conflict**: Dataflow §1.1 pending output shows `isInitialLoading:true` but omits `isLoading`; UC-11 says it's also `true`. Dataflow §1.4 refreshing output shows `isRefreshing:true` but omits `isLoading`; UC-11 says it's also `true`. A reader relying on the dataflow alone might incorrectly infer that `isLoading` is `false` or undefined in these states.

**Suggested fix**: Standardize dataflow diagram state outputs to always include `isLoading` alongside other boolean flags, or add a note referencing UC-11's complete table.

### #4 — UC-4 comment: "resource.query()" should be "entry.query()"

**Severity**: Low
**Usecases**: UC-4 imperative retry:
```typescript
agent.start({ id: "1" }); // resource.query() sees error state → fetches again
```

**Dataflow**: §1.6 Error → Retry sequence:
```
App → Agent: start(args)
Agent → Agent: _getEntry(args)
Agent → RCE: query()
```

The agent calls `_getEntry(args)` (factory callback from ResourceV2), then calls `entry.query()` on the returned `ResourceV2CacheEntry`. It does NOT call `resource.query(args)` — that is a separate public API used in UC-12 server-side (`await userResource.query({id:"1"})`).

**Fix**: UC-4 comment should say `entry.query()` instead of `resource.query()`.
