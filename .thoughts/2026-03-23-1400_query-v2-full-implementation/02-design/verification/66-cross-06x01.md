## Issues Found

### Issue #1 — Plugins test section mislabeled as "core Layer"

- **What's wrong**: Test section is titled "Test Cases — core Layer: Plugins" (PL01–PL08). Architecture §2 explicitly defines plugins/ as **Layer 5**, a separate layer above react (Layer 4). Plugins are NOT part of the core layer.
- **Where**: 06-testcases.md, section header before PL01
- **Expected**: "Test Cases — plugins Layer" or "Test Cases — plugins Layer: ReactHooksPlugin"
- **Severity**: Medium — misrepresents the architectural layering; could lead to incorrect test file placement during implementation

### Issue #2 — Snapshot tests not grouped under core layer

- **What's wrong**: Tests use standalone header "Test Cases — Snapshot" (SN01–SN12). Architecture §3 places Snapshot in the core layer "Infrastructure" subsystem alongside LifecycleHooks.
- **Where**: 06-testcases.md, section header before SN01
- **Expected**: "Test Cases — core Layer: Snapshot" (consistent with other core subsections like "core Layer: CacheEntry", "core Layer: LifecycleHooks")
- **Severity**: Low — organizational inconsistency; Snapshot IS core but presented as standalone

### Issue #3 — `initialSnapshot` on `createApi` not in architecture

- **What's wrong**: Tests AP08, AP08a, AP08b, AP08c, SN07, SN09, SN11 reference `createApi({initialSnapshot})` as a configuration pattern. Architecture §7.1 lists `hydrateSnapshot()` as a standalone public API but does not describe `initialSnapshot` as a `createApi` option. The per-resource "save → consume slice → delete slice" lifecycle is test-only knowledge.
- **Where**: 06-testcases.md, AP08 cluster and SN07/SN09/SN11
- **Expected**: Architecture §3a or §7.1 should describe `initialSnapshot` option on `createApi` and the per-resource consumption pattern, OR tests should align with the `hydrateSnapshot()` standalone API if that is the intended design.
- **Severity**: Low — likely an architecture abstraction gap rather than a test error; the pattern is well-described in tests and consistent with ADR/dataflow documents

### Issue #4 — `_savedSnapshot` internal field referenced in tests, absent from architecture

- **What's wrong**: Tests AP05, AP08b, SN11, SN12 reference `_savedSnapshot` as an internal field on the API instance. Architecture §3a diagrams enumerate `_resources: Set<ResourceV2>` and `Merged Configuration` as createApi internals but omit `_savedSnapshot`.
- **Where**: 06-testcases.md, AP05/AP08b/SN11/SN12
- **Expected**: Architecture §3a should include `_savedSnapshot` in the createApi Factory diagram if tests depend on its lifecycle behavior
- **Severity**: Low — implementation detail; tests can be more specific than architecture

### Issue #5 — Type-level tests promised but no test case IDs

- **What's wrong**: Plugin Testing Strategy (line ~100) states "Verify PluginAugmentations types compile correctly (type-level tests)" but no concrete SM/PL/RH test case ID covers type-level compilation verification.
- **Where**: 06-testcases.md, Plugin Testing Strategy section
- **Expected**: Either add explicit test cases (e.g., `PL09: PluginAugmentations<ReactHooksPlugin, TArgs, TData> resolves to correct type`) or clarify that type-level tests are implicit in .test-d.ts files outside the ID scheme
- **Severity**: Low — type-level tests are typically handled differently (tsd, expect-type) and may not need formal IDs, but the promise-without-delivery is a minor gap

### Issue #6 — `createCacheMap()` factory not in architecture

- **What's wrong**: Tests CM-F04 and CM-F05 reference `createCacheMap({ keyStrategy: "serialize" | "compare" })` as a factory function that returns either `SerializeCacheMap` or `CompareCacheMap`. Architecture §5.2 shows `ICacheMap` interface with two implementations but does not mention a `createCacheMap()` factory function.
- **Where**: 06-testcases.md, CM-F04/CM-F05; architecture §5.2
- **Expected**: Either architecture §5.2 should mention the factory, or tests should construct `SerializeCacheMap` / `CompareCacheMap` directly as the architecture implies
- **Severity**: Low — reasonable implementation detail for a factory pattern; not a semantic inconsistency
