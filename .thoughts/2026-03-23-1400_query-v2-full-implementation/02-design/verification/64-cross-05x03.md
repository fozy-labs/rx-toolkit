## Findings

### Finding #1 ‚Äî API-level `compareArg` type mismatch
**Severity**: Medium
**Primary**: UC-16 in 05-usecases.md
**Comparison**: ¬ß12.1 `ICreateApiOptions` in 03-model.md
**What's wrong**: UC-16 passes a typed comparator at the API level:
```typescript
const api = createApi({
    compareArg: (a: RegExp, b: RegExp) => a.source === b.source && a.flags === b.flags,
});
```
But the model's `ICreateApiOptions` defines `compareArg?: TCompareArgsFn` without a generic parameter, resolving to `TCompareArgsFn<unknown>` = `(a: unknown, b: unknown) => boolean`. Under TypeScript strict function types, `(a: RegExp, b: RegExp) => boolean` is **not assignable** to `(a: unknown, b: unknown) => boolean` (parameter contravariance). The resource-level `IResourceV2Options<TArgs, TData>` correctly parameterizes as `TCompareArgsFn<TArgs>`, but the API-level options lack a `TArgs` generic entirely.
**What's expected**: Either (a) the use case should set `compareArg` at the resource level where it IS parameterized, or (b) the model's `ICreateApiOptions` needs a looser type signature for `compareArg` (e.g., `(a: any, b: any) => boolean`), or (c) `ICreateApiOptions` gains a `TArgs` parameter (unlikely since API serves heterogeneous resources).
**Impact**: UC-16 code example would fail TypeScript strict mode compilation.
