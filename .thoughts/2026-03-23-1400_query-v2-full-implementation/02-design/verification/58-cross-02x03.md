## Consistent Areas

The following areas were cross-checked and found consistent between both files:

- **Machine state shapes**: Dataflow §4.1 "State data availability" table matches Model §2 state types exactly (`data`, `error`, `args`, patches-eligible fields all align).
- **Generic parameters `<TArgs, TData>`**: Both files consistently use `<TArgs, TData>` throughout. No `TError`. Data-only types correctly use `<TData>` alone.
- **`queryFn` signature**: Dataflow `queryFn(args: TArgs, {abortSignal: AbortSignal}): Promise<TData>` matches Model §7.1 `TQueryFn<TArgs, TData>`.
- **Agent `_getEntry` callback**: Dataflow §1.1 factory callback description matches Model §8.1 `_getEntry: (args: TArgs) => ResourceV2CacheEntry<TArgs, TData>`.
- **Agent `_tracking$` signal**: Dataflow §1.1 `_tracking$` usage matches Model §8.1 `_tracking$: SignalFn<{previous, current}>`.
- **`IResourceV2AgentState` fields**: All fields shown in dataflow diagrams (`status`, `data`, `isLoading`, `isInitialLoading`, `isRefreshing`) are present in Model §8.1.
- **CacheMap interface**: Dataflow `CacheMap.getOrCreate(args)`, `.get(args)`, `.delete(args)` match Model §6.1 `ICacheMap`.
- **CacheEntry.complete()**: Dataflow §1.7/§5.4 usage matches Model §5 definition.
- **Plugin lifecycle**: Dataflow §3.1–3.4 `install(context: IPluginContext)` and `augmentResource()` match Model §11 plugin types.
- **Snapshot types and lifecycle**: Dataflow §2.1 three-phase `_savedSnapshot` lifecycle matches Model §12.1 description exactly.
- **RCE abort management**: Dataflow §1.5 per-entry `AbortController` matches Model §7.3 `_abortController: AbortController | null`.
- **ADR-2 refreshing error behavior**: Dataflow §1.4 error-during-refresh stays in Success matches Model §2 `TRefreshingState` → `TSuccessState` path.
- **`IPatchHandle`**: Dataflow §5.5 `createPatch` return type matches Model §3 `IPatchHandle`.
- **snapshot `Machine.fromSnapshot()`**: Dataflow §2.1 matches Model §3 `IMachineStatic.fromSnapshot<TArgs, TData>()`.

## Verdict: INCONSISTENT

Seven inconsistencies found. Issues #1 and #2 are **high severity** — they affect machine API completeness and state machine correctness respectively. Issues #3 and #4 are **medium severity** — missing internal class definition leaves dataflow's internal references ungrounded in the model. Issues #5, #6, and #7 are **low severity** — missing internal infrastructure types that are referenced in dataflow but absent from/underdefined in the model.
