## Findings

### F-67-01 — CRITICAL: Abort-on-args-change contradicts dataflow §1.5

**Tests:** RE11 ("Abort: args change cancels inflight request — query({id:1}) then query({id:2}) → First request's AbortSignal aborted"), INT12 ("Abort signal propagated to queryFn on args change — Agent.start(args1) → agent.start(args2) → args1 queryFn receives aborted signal")

**Dataflow §1.5 note:** "When an agent switches from args1 to args2, it obtains a new entry for args2 and starts a query on it. The old entry's (args1) inflight request **continues independently** — it is not aborted by the agent. Other consumers (agents, hooks) still using the args1 entry will receive its data normally."

**Conflict:** RE11 and INT12 assert the old request IS aborted on args change. The dataflow explicitly says it is NOT — abort only happens within the **same** entry (invalidation / force re-fetch). These are mutually exclusive behaviors.

**Resolution needed:** Decide which is correct. The dataflow rationale (other consumers may still need args1 data) is architecturally sound — shared cache entries should NOT be aborted by one agent switching away. Tests RE11 and INT12 should be corrected to match, OR the dataflow note should be removed if the design intent is per-agent abort.

### F-67-02 — MEDIUM: No test for GC with inflight fetch in progress

**Dataflow §1.7 (GC cleanup decisions):** "Abort inflight fetch — YES: When GC fires, refcount has been 0 for cacheLifetime, but a fetch may still be in-flight (scenario: component starts fetch → unmounts → GC timer fires before fetch resolves). ... Calling _abortController.abort() prevents wasted network requests."

**Dataflow §5.4 (GC Trigger Flow):** Shows explicit "abort inflight fetch if any" step between timer expiry and complete().

**Tests:** GC01–GC05 cover timer start/cancel/disable/fire, but none test the scenario where the GC timer fires while a `queryFn` call is still in-flight. No test verifies the inflight request is aborted. RCE15 tests `complete()` generically but focuses on patches/onClean$, not inflight abort. E06 tests resetCache() during inflight — different path.

**Expected test:** GC scenario: start fetch → unmount (refcount→0) → advance timer past cacheLifetime → verify queryFn's AbortSignal is aborted → verify entry cleaned up.

### F-67-03 — MEDIUM: No test for `cacheLifetime: 0` (immediate GC)

**Dataflow §1.7 table:** `cacheLifetime` value `0 or negative` → `resetOnRefCountZero: true` → "Immediate reset on last unsubscribe."

**Tests:** GC03 covers `cacheLifetime: false` (disabled). GC01 covers generic timer. No test specifically verifies `cacheLifetime: 0` triggers immediate cleanup on unsubscribe without any timer delay.

**Expected test:** Create resource with `cacheLifetime: 0` → subscribe → unsubscribe → entry immediately deleted (no timer advance needed).

---

### F-67-04 — LOW: `complete()` patch behavior — test vs dataflow §1.7 nuance

**Test RCE15:** "`entry.complete()` is terminal: aborts patches → idle → onClean$ → completed" — expects "Pending patches aborted" as part of `complete()`.

**Dataflow §1.7 (GC cleanup):** "Abort patches — NO: When refcount reaches 0 and the GC timer fires, no subscribers exist. ... pending patches are impossible. No abort needed."

**Not a hard contradiction:** `complete()` may defensively abort patches, and GC simply never encounters them. However the dataflow says "No abort needed" which could be read as "complete() should not include abort logic." The design should clarify: does `complete()` unconditionally abort patches (defensive), or is patch abort only in `resetCache()` (explicit)? If defensive, the dataflow §1.7 note should say "not needed for GC, but complete() handles it generically."

### F-67-05 — LOW: No test for later plugin seeing earlier plugin's contributions (§3.2)

**Dataflow §3.2:** "Plugins are applied sequentially. Later plugins can see earlier plugins' contributions (they're already on the resource object)."

**Tests:** PL04 verifies installation ORDER. PL03 verifies Object.assign merge. No test verifies that plugin B's `augmentResource()` can access plugin A's contribution on the resource object. This is a stated dataflow behavior without test coverage.

### F-67-06 — LOW: Error state `data=null` not explicitly asserted

**Dataflow §4.1 (State data availability table):** Error state has `data: null`.

**Tests:** SM06 asserts `instanceof MachineError`, `status="error"`, `error is Error`. SM19 asserts error preserves error value. Neither explicitly asserts `data === null`. The table is an authoritative specification — each cell should have a corresponding assertion.

---

### F-67-07 — LOW: No test for `getEntry$` binding optimization (§6.2)

**Dataflow §6.2:** Describes specific binding behavior: `if (binded) return binded; entry = _lastEntry$(); if (entry.isMyArgs(args)) binded = entry; return entry`. This is a caching optimization inside the computed signal.

**Tests:** RE08 (reactive to resetAll) and RE23 (returns null when idle) test observable behavior but not the binding memo. No test verifies that once `getEntry$` binds to an entry, it returns the same reference on subsequent reads without re-querying `_lastEntry$`.

**Impact:** Minor — the binding is an implementation optimization. Observable behavior is covered.
