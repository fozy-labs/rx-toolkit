## Cross-Check Results

### XC-1 — Plugin type augmentation: PluginAugmentations (ADR-9) vs declare module
**Status**: PASS
**ADR-9 Decision**: Use `PluginAugmentations<TPlugin, TArgs, TData>` generic conditional type augmentation. Explicitly rejects `declare module` as legacy anti-pattern.
**UC-10 Evidence**: Shows full `PluginAugmentations` resolution chain:
- `resource: IResourceV2<{id: string}, User> & PluginAugmentations<readonly [ReactHooksPlugin], {id: string}, User>`
- `PluginResourceContributions<TPlugin, TArgs, TData>` conditional type
- `UnionToIntersection` utility for intersecting contributions
- Zero `declare module` usage anywhere in usecases.

**UC-14 Evidence**: Multi-plugin composition also uses `PluginAugmentations`, extending via conditional type branch — not `declare module`. Custom plugin type extension shown as adding a conditional type branch to `PluginResourceContributions`, not a `declare module` block.

### XC-2 — resetAll follows ADR-16 single API instance
**Status**: PASS
**ADR-16 Decision**: `createApi()` is the canonical entry point. `resetAll()` is a method on `IApi`, coordinating across all tracked resources.
**UC-13 Evidence**: `api.resetAll()` is called on the API instance. Steps enumerate: `_savedSnapshot = null`, then iterates `_resources: Set<ResourceV2>`, calling `CacheEntry.complete()` on all entries. No standalone `resetAllCacheV2()` function. No global-state reset.
**UC-12 Step 3**: `clientApi.resetAll()` — also called on the API instance, consistent with ADR-16.

### XC-3 — SSR use case follows ADR-8 snapshot bridge lifecycle
**Status**: PASS
**ADR-8 Decision**: Three-phase hydration: (1) `createApi({ initialSnapshot })` saves snapshot as `_savedSnapshot`, (2) each `createResourceV2()` consumes its slice from `_savedSnapshot` and deletes it, (3) `resetAll()` deletes `_savedSnapshot` entirely.
**UC-12 Evidence**: All three phases explicitly documented:
1. Server: `serverApi.getSnapshot()` → extracts `.state` from `MachineSuccess` entries only (matches ADR-8 capture flow)
2. Client: `createApi({ initialSnapshot })` — "snapshot is saved but NOT hydrated yet" (phase 1)
3. Client: `createResourceV2()` — "consumes snapshot slice for 'users' key... DELETED from _savedSnapshot" (phase 2), `Machine.fromSnapshot<TArgs, TData>(slice)` for reconstruction
4. Client: `clientApi.resetAll()` — "_savedSnapshot = null" (phase 3)

### XC-4 — SSR follows ADR-12 no structural sharing

No mention of structural sharing, reference reuse, or deep-equal comparison. Additionally, `maxSnapshotDataAge` check triggers auto-invalidation when stale — matching ADR-12's staleness handling.

### XC-6 — SWR previous/current swap follows ADR-3
**Status**: PASS
**ADR-3 Decision**: Keep previous entry until current resolves. Previous nulled only on current success/error. Rapid changes: don't chain previous.
**UC-3 Evidence**: Shows SWR transition: "userId changes '1' → '2': status = 'pending', data = user1Data (SWR — previous entry's data)" → "After fetch: status = 'success', data = user2Data". Previous data shown during loading.
**UC-11 State Table**: Confirms SWR row: "Args change (SWR)" → `status="pending"`, `data=User (prev)`, `isLoading=true`, `isInitialLoading=false`.
**UC-2 Edge Case**: Rapid arg changes described: "Agent tracks only the latest `current` entry. Inflight requests for entries '1' and '2' continue independently" — no chaining of previous (matches ADR-3 point 2).

### XC-7 — GC follows ADR-5 refcount + timer hybrid
**Status**: PASS
**ADR-5 Decision**: Refcount + timer hybrid via `share({resetOnRefCountZero})`. GC timer starts only when refcount reaches 0.
**UC-7 Evidence**: "useResourceV2Agent creates agent, subscribes to signal → refcount increments. On unmount: agent destroyed, signal unsubscribed → refcount decrements. If refcount reaches 0: GC timer starts (30_000ms). If remounted before timer: GC cancelled, cached data shown instantly. If timer fires: CacheEntry.complete() → entry removed." — exact match with ADR-5 behavior.

### XC-8 — Agent.start() behavior follows ADR-10 (query on first, observe on subsequent)
**Status**: PASS
**ADR-10 Decision**: `start()` obtains entry via `_getEntry` callback and calls `entry.query()`, which internally deduplicates (returns existing promise if fresh/inflight).
**UC-1 Evidence**: `agent.start()` triggers fetch — "void args — no argument needed".
**UC-2 Evidence**: `agent.start({ id: "1" })` → "fetches user 1", then `start({ id: "2" })` → "fetches user 2, triggers SWR".
**UC-9 Evidence**: Two components both call `useResourceV2Agent(userResource, { id: "1" })` — "Only ONE fetch... inflight dedup in ResourceV2.query". Dedup behavior matches ADR-10's entry-level dedup.

### XC-9 — Abort management at CacheEntry level (ADR-17)
**Status**: PASS
**ADR-17 Decision**: Each `ResourceV2CacheEntry` manages its own `AbortController`. No resource-level `_inflightMap`.
**UC-1 Evidence**: `queryFn` receives `{ abortSignal }` — abort is per-entry.
**UC-2 Edge Case**: "Each `ResourceV2CacheEntry` manages its own `AbortController`, aborting the previous inflight only when a new `query()` call is made to the same entry." — explicitly per-entry, no resource-level abort coordination.
**UC-9 Dedup**: "Each `ResourceV2CacheEntry` tracks its own inflight promise. When `entry.query()` is called while a request is already in flight for that entry, it returns the existing promise" — matches ADR-17 entry-level inflight tracking.

### XC-10 — Agent independence from Resource (ADR-18)
**Status**: PASS
**ADR-18 Decision**: Agent receives `_getEntry` and `_compareArgs` callbacks. No direct Resource reference.
**UC-17 Evidence**: "Agent calls `_getEntry(args)` callback → ResourceV2 delegates to `cache.getOrCreate(args)` → entry returned → Agent calls `entry.query()`". Agent-to-entry flow via callback, never via Resource.
**UC-2 Edge Case**: "each `start()` call obtains an entry via the factory callback" — factory callback pattern, not direct Resource method call.

### XC-11 — CacheMap dual implementation (ADR-19) in use cases
**Status**: PASS
**ADR-19 Decision**: `ICacheMap` interface + `SerializeCacheMap`/`CompareCacheMap` + `createCacheMap` factory. Factory callback for entry creation.
**UC-16 Evidence**: "Internally: `CompareCacheMap.getOrCreate(/foo.*bar/i)` → Linear scan over `_entries` using `compareArg`" — matches `CompareCacheMap` with array + linear scan.
**UC-17 Evidence**: Shows `createCacheMap(options)` factory with `factory: (args: TArgs) => new ResourceV2CacheEntry(...)`, `keyStrategy`, `serializeArgs`, `compareArg`. Matches ADR-19 exactly.

### XC-12 — Patcher consistency violation (ADR-6) in use cases
**Status**: PASS
**ADR-6 Decision**: Patcher returns `IPatchResolution` with `TPatchState`. On consistency violation, `isConsistencyViolation` flag set, `ResourceV2CacheEntry` auto-invalidates.
**UC-5 Edge Case**: Multi-patch out-of-order abort shows: "Patcher detects consistency violation → ResourceV2CacheEntry auto-invalidates. isConsistencyViolation is tracked in TPatchState on the machine state. Cache shows last valid patched data until fresh data arrives from refetch." — matches ADR-6 decision. The `[ref: ADR-6]` link is explicit.

### XC-13 — State machine immutable class-based (ADR-2) in use cases

**UC-6 Note**: "Only `MachineSuccess` entries can transition to `MachineRefreshing`" — consistent with ADR-2's state transition model.
