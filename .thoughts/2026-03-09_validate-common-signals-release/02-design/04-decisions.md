# 04 — Архитектурные решения (ADR)

## ADR-1: Выбор тестового фреймворка {#adr-1}

**Status**: Принято  
**Дата**: 2026-03-09  
**Решающий**: Человек (ответ на [Q1](../01-research/04-open-questions.md#q1))

### Контекст

В проекте полностью отсутствует тестовая инфраструктура ([03-constraints](../01-research/03-constraints.md#21-отсутствие-тестовой-инфраструктуры)). Проект использует:
- `"type": "module"` (ESM)
- `"moduleResolution": "bundler"` (Vite-стиль)
- Path aliases `"@/*": ["src/*"]`
- Vite в `apps/demos/`

### Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Vitest** | Нативный ESM, Vite-экосистема, быстрый, path aliases из коробки | Менее зрелый чем Jest |
| **Jest** | Зрелая экосистема, широкое сообщество | ESM экспериментальный, сложная настройка path aliases |

### Решение

**Vitest**

### Обоснование

1. ESM-first проект — Vitest нативно поддерживает ESM ([02-external-research](../01-research/02-external-research.md#4-выбор-тестового-фреймворка))
2. Vite уже в проекте (`apps/demos/vite.config.ts`) — единая экосистема
3. Path aliases через `resolve.alias` в `vitest.config.ts` — без дополнительной настройки
4. TypeScript через esbuild — без `ts-jest` или `@babel/preset-typescript`

### Последствия

- (+) Минимальная конфигурация
- (+) Быстрый запуск тестов
- (+) API совместим с Jest (переход возможен)
- (-) Для CI может потребоваться установка Vitest глобально или через npx

## ADR-2: Scope — тесты + критические фиксы {#adr-2}

**Status**: Принято  
**Дата**: 2026-03-09  
**Решающий**: Человек (ответ на [Q3](../01-research/04-open-questions.md#q3))

### Контекст

Исследование выявило множество потенциальных проблем ([01-codebase-analysis](../01-research/01-codebase-analysis.md)):
- `Batcher.run()` без `try/finally` — блокировка системы при любой ошибке
- `deepEqual` не обрабатывает `Date`, `RegExp`, `NaN`, циклические ссылки
- `reduxDevtools` throws при отсутствии расширения
- `LocalState.DEFAULT_DRIVER = localStorage` — ошибка в SSR
- `useSignal` без `getServerSnapshot`

### Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Только тесты | Минимальный риск | Оставляет критические баги |
| Тесты + все фиксы | Полное исправление | Большой scope, риск breaking changes |
| **Тесты + критические фиксы** | Баланс | Нужно определить критерий "критичности" |

### Решение

**Тесты + критические фиксы**

### Критерий критичности

Фикс считается критическим, если:
1. Баг может заблокировать всю систему (Batcher try/finally)
2. Баг приводит к uncaught exception в production (НЕ edge case)

### Классификация найденных багов

| Баг | Критичность | Действие |
|-----|-------------|----------|
| `Batcher.run()` без try/finally | **КРИТИЧЕСКИЙ** | Фикс + тест |
| `deepEqual` неполный | Низкая | Документация + `.skip` тесты |
| `reduxDevtools` throw | Средняя | Документация (SSR не в scope) |
| `LocalState.DEFAULT_DRIVER` | Средняя | Документация (SSR не в scope) |
| `useSignal` без `getServerSnapshot` | Средняя | Документация (SSR не в scope) |
| `JSON.parse` без try/catch в LocalState | Средняя | Документация + тест |

### Последствия

- (+) Критический баг Batcher исправлен — система не заблокируется при ошибках
- (+) Некритические баги задокументированы через тесты
- (-) Часть известных проблем остаётся нерешённой
- (-) Нужно аккуратно пометить, какие тесты document behavior vs test correctness

## ADR-3: Стратегия изоляции тестов для глобальных синглтонов {#adr-3}

**Status**: Принято  
**Дата**: 2026-03-09

### Контекст

Проект содержит 4 глобальных синглтона ([03-constraints](../01-research/03-constraints.md#31-глобальное-состояние-синглтоны)):
- `Scheduled` (Batcher.ts) — `map`, `lowestRang`, `isLocked`
- `SharedOptions` — `DEVTOOLS`, `onQueryError`
- `Indexer` — `currentIndex`
- `DependencyTracker` — `_currentHandler`

Ни один из них не имеет метода `reset()`. Изменение исходного кода для добавления reset-методов нежелательно на этом этапе (scope: минимальные фиксы).

### Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Добавить reset() во все синглтоны | Чистоизоляция | Модификация API, расширение scope |
| `vi.resetModules()` в каждом тесте | Полная изоляция | Медленно, сложно |
| **`pool: 'forks'` + beforeEach** | Изоляция между файлами + частичная внутри | Быстро, без модификации кода |
| Worker threads | Максимальная изоляция | Оверхед, сложная настройка |

### Решение

**Двухуровневая изоляция**: `pool: 'forks'` + `beforeEach/afterEach`

### Детали

1. **Уровень файла**: Vitest `pool: 'forks'` — каждый тестовый файл в отдельном процессе. Это полностью изолирует `Scheduled`, `Indexer` и другие модульные синглтоны.

2. **Уровень теста**: `beforeEach` для сброса доступных полей:
   - `SharedOptions.DEVTOOLS = null` — сбрасываем
   - `DependencyTracker` — проверяем что `_currentHandler === null` (утечка tracked context)

3. **Валидация**: `afterEach` проверяет, что `Scheduled.isLocked === false` (после фикса Batcher). Если locked — тест оставил систему в некорректном состоянии.

### Последствия

- (+) Не нужно модифицировать исходный код
- (+) `pool: 'forks'` даёт полную изоляцию между файлами
- (-) Тесты внутри одного файла делят состояние `Indexer.currentIndex` — но это не влияет на корректность, только на конкретные значения индексов
- (-) `pool: 'forks'` медленнее `pool: 'threads'` — приемлемо для ~20 тестовых файлов

## ADR-4: Обработка deprecated API {#adr-4}

**Status**: Принято  
**Дата**: 2026-03-09  
**Решающий**: Человек (ответ на [Q2](../01-research/04-open-questions.md#q2))

### Контекст

В scope 5 deprecated API ([03-constraints](../01-research/03-constraints.md#41-deprecated-api-в-scope)):
- `Signal` constructor → `State` constructor
- `Signal.create()` → `Signal.state()`
- `Effect.complete()` → `Effect.unsubscribe()`
- `validator$` option → `checkEffect`
- `LocalSignal` → `LocalState`

### Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Удалить deprecated API | Чище codebase | Breaking change |
| **Оставить + пометить для v0.6.0** | Обратная совместимость | "Мертвый" код в тестах |
| Игнорировать deprecated | Меньше тестов | Может сломаться незаметно |

### Решение

**Оставить deprecated API, тестировать, пометить для удаления в v0.6.0**

### Детали

Каждый deprecated API получает:
1. Тест, проверяющий функциональность (`it('deprecated: ...')`)
2. Тест, проверяющий что API эквивалентен замене
3. Комментарий `// TODO(v0.6.0): remove deprecated API`

### Последствия

- (+) Гарантия обратной совместимости для текущих пользователей
- (+) Ясная метка для будущего удаления
- (-) Дополнительные тесты для deprecated кода
- (-) Усложняет test suite (дублирование тестов old/new API)

## ADR-5: deepEqual — scope фикса {#adr-5}

**Status**: Принято  
**Дата**: 2026-03-09

### Контекст

`deepEqual` имеет ряд известных ограничений ([01-codebase-analysis](../01-research/01-codebase-analysis.md)):
- Не обрабатывает `Date`, `RegExp`, `Map`, `Set`
- Не обрабатывает `NaN`
- Не обрабатывает циклические ссылки
- Использует `// @ts-ignore`

### Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Полный фикс | Корректный deepEqual | Большой scope, потенциальные breaking changes |
| Заменить на библиотеку | Проверенная реализация | Новая зависимость |
| **Документировать ограничения** | Минимальный risk | Баги остаются |

### Решение

**Только документация через тесты**

### Обоснование

1. `deepEqual` не используется в `src/signals/` — только в `src/common/utils/`
2. В текущем scope (signals + common) `deepEqual` не вызывается критическими путями
3. `shallowEqual` используется в `SharedOptions.defaultCompareArgs`, но только для `src/query` (не в scope)
4. Полный фикс `deepEqual` — отдельная задача с собственным анализом совместимости

### Детали

- Рабочие случаи покрываются обычными тестами
- Известные ограничения покрываются `it.skip(...)` с комментарием
- `@ts-ignore` оставляется как есть (вне scope критических фиксов)

### Последствия

- (+) Нет риска breaking changes
- (+) Ограничения задокументированы для будущих разработчиков
- (-) `deepEqual` остаётся неполным
