## Детальное описание рисков

### R1: Breaking changes от фикса Batcher {#risk-1}

**Описание**: Фикс `Batcher.run()` (добавление try/finally) изменяет поведение при ошибках. Текущее поведение: система блокируется навсегда. Новое поведение: система восстанавливается.

**Вероятность**: Низкая — фикс строго улучшает поведение. Никто не зависит от "залоченного" состояния.

**Влияние**: Высокое — если фикс некорректен, может нарушить порядок батчинга.

**Митигация**:
1. Фикс — одна строка: обернуть в `try/finally`
2. Тест до фикса: подтвердить сломанное поведение
3. Тест после фикса: подтвердить восстановление
4. Integration test: сложный сценарий с ошибкой в середине батча

**Статус**: Контролируемый ✅

---

### R2: Flaky тесты из-за async/timing {#risk-2}

**Описание**: Код использует `queueMicrotask` (useSignal, reduxDevtools) и `setTimeout` (batching strategies). Тесты, зависящие от timing, могут быть нестабильными.

**Вероятность**: Средняя — microtask timing детерминирован в Node.js, но React concurrent mode добавляет неопределённость.

**Влияние**: Среднее — flaky тесты подрывают доверие к test suite.

**Митигация**:
1. Утилита `flushMicrotasks()` — явное ожидание microtask queue
2. `act()` из `@testing-library/react` — для React-тестов
3. `vi.useFakeTimers()` — для тестов с `setTimeout`
4. Конвенция: все async тесты помечены `async` и используют утилиты

**Проблемные файлы**:
- `useSignal.ts` — `queueMicrotask` для дебаунса ([01-codebase-analysis](../01-research/01-codebase-analysis.md))
- `reduxDevtools.ts` — `setTimeout(0)` в `BatchStrategy.Async`

**Статус**: Контролируемый при соблюдении конвенций ✅

---

### R3: Утечка глобального состояния между тестами {#risk-3}

**Описание**: 4 глобальных синглтона ([03-constraints](../01-research/03-constraints.md#31-глобальное-состояние-синглтоны)) не имеют reset-методов. Состояние из одного теста может "утечь" в следующий.

**Вероятность**: Высокая — без специальных мер это произойдёт в каждом файле.

**Влияние**: Высокое — тесты будут проходить/падать в зависимости от порядка выполнения.

**Митигация** (см. [ADR-3](./04-decisions.md#adr-3)):
1. `pool: 'forks'` — изоляция между файлами (fresh process)
2. `beforeEach` — reset доступных полей (`SharedOptions`)
3. `afterEach` — assertion что система в чистом состоянии
4. Тесты НЕ зависят от конкретных значений `Indexer.currentIndex`

**Остаточный риск**: Тесты внутри одного файла делят `Scheduled` state. При ошибке в тесте (до фикса Batcher) `Scheduled.isLocked` может остаться `true`. После фикса Batcher этот риск устранён.

**Статус**: Контролируемый после фикса Batcher ✅

### R4: SSR-тесты без среды

**Описание**: `reduxDevtools.ts` использует `window`, `LocalState.ts` использует `localStorage`, `useSignal.ts` не предоставляет `getServerSnapshot`. SSR-совместимость не в scope ([Q4](../01-research/04-open-questions.md#q4)).

**Вероятность**: Низкая — SSR явно исключён.

**Влияние**: Низкое — пользователи SSR получат ожидаемые ошибки.

**Митигация**: Документировать ограничения в `CHANGELOG.md` и/или JSDoc.

**Статус**: Принятый риск ✅

---

### R5: Deprecated API — потраченное время

**Описание**: 5 deprecated API требуют тестов, которые станут неактуальными при удалении в v0.6.0.

**Вероятность**: Низкая — тесты deprecated API минимальны.

**Влияние**: Низкое — < 5% от общего объёма тестов.

**Митигация**: Минимальные тесты, группировка в `describe('deprecated')`, метки `TODO(v0.6.0)`.

**Статус**: Принятый риск ✅

---

### R6: Циклические зависимости ломают тестовые imports {#risk-6}

**Описание**: Исследование выявило 2 цикла зависимостей ([03-constraints](../01-research/03-constraints.md#32-циклические-зависимости)):
1. `common/reduxDevtools → signals/Batcher` (межмодульный)
2. `signals/Computed ↔ signals/Signal` (внутримодульный)

При тестировании изолированных модулей циклические imports могут привести к `undefined` при загрузке.

**Вероятность**: Средняя — ESM с lazy evaluation обычно выдерживает циклы, но в тестовой среде порядок импортов может отличаться.

**Влияние**: Высокое — тесты не запустятся.

**Митигация**:
1. Тестировать через публичный API (`Signal.state()`, `Signal.compute()`) — обходя прямые импорты
2. Для unit-тестов внутренних модулей (`Batcher`, `ComputeCache`) — импортировать напрямую из файла, не через barrel exports
3. Если цикл ломает import — использовать `vi.importActual()` или `vi.mock()` для разрыва

**Статус**: Контролируемый ✅

---

### R7: Coverage < 80%

**Описание**: Некоторые code paths сложно покрыть: `FinalizationRegistry` callback, race conditions, error handling в RxJS operators.

**Вероятность**: Средняя — 80% реалистичен, но потребует целенаправленных усилий.

**Влияние**: Среднее — формальный порог не достигнут, но ключевые пути покрыты.

**Митигация**:
1. Приоритизация: сначала публичный API (State, Computed, Effect)
2. Сложные paths: FinalizationRegistry, SyncObservable edge cases — документировать как исключения из coverage
3. Coverage exclude для заведомо нетестируемых путей (devtools integration)

**Статус**: Контролируемый ✅
