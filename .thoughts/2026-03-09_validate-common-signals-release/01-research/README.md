# Исследование: validate-common-signals-release

**Status**: Draft  
**Дата**: 2026-03-09  
**Scope**: `src/common/` и `src/signals/` (без `src/query`)  
**Цель**: Валидация кода для публичного релиза — написание тестов, поиск ошибок, проверка качества

## Контекст

Библиотека `@fozy-labs/rx-toolkit` (v0.5.3-rc.2) — набор инструментов для реактивного управления состоянием на основе RxJS. Перед публичным релизом необходимо провести аудит качества кода модулей `src/common/` и `src/signals/`.

## Ключевые находки

1. **Тесты полностью отсутствуют** — в проекте нет ни одного тестового файла, тестового фреймворка или скрипта для запуска тестов
2. **Циклическая зависимость common ↔ signals** — `reduxDevtools.ts` импортирует `Batcher` из `@/signals`, а `signals` импортирует из `common`
3. **Циклические зависимости внутри signals** — `Computed → Signal → Computed` через barrel exports
4. **4 `@deprecated` API** в исследуемом scope без плана удаления
5. **`useSignal` — потенциальные edge-cases** с `queueMicrotask` оптимизацией и `useSyncExternalStore`
6. **`SyncObservable.value`** — subscribe/unsubscribe в одном стеке может быть хрупким паттерном
7. **`Batcher.Scheduled.run()` — рекурсия** без защиты от переполнения стека

## Топ-3 критических вопроса

1. **Выбор тестового фреймворка** (Vitest vs Jest) — блокирует всю работу по тестированию
2. **Как обрабатывать цикл `common ↔ signals`** — это архитектурный вопрос, влияющий на тестируемость
3. **Судьба deprecated API** — удалять перед релизом или оставить с предупреждениями?

## Рекомендация

Перед фазой Design необходимо решить вопросы **#1** и **#3**. Вопрос **#2** может быть решён в рамках дизайна, но требует осведомлённого решения.
