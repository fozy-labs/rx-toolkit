# 04 — Открытые вопросы

## Критические (блокируют Design)

### Q1: Выбор тестового фреймворка
**Контекст**: В проекте нет никакой тестовой инфраструктуры. Проект использует ESM, Vite (в demos/), TypeScript с path aliases.

**Варианты:**
- **Vitest** — нативная ESM поддержка, совместимость с Vite, быстрый, поддержка path aliases из vite.config
- **Jest** — более зрелый, огромная экосистема, но ESM-поддержка экспериментальная и path aliases требуют отдельной настройки

**Рекомендация исследователя**: Vitest. Основания: ESM-first проект, Vite уже в проекте, меньше конфигурации.

**Нужно решение человека**: Да / Нет / Другой фреймворк

Ответ: vitest

---

### Q2: Что делать с deprecated API перед релизом?

**Контекст**: В scope есть 5 deprecated API (см. [03-constraints.md](./03-constraints.md#41-deprecated-api-в-scope)).

**Варианты:**
1. **Оставить как есть** — добавить тесты и для deprecated API, обеспечить обратную совместимость
2. **Удалить deprecated API** — breaking change, но чище кодовая база
3. **Оставить, но пометить для удаления в конкретной версии** (например, v0.6.0)

**Нужно решение человека**: Выбор стратегии

Ответ: Оставить, но пометить для удаления в конкретной версии (Если я проавильно помню, часть api уже помечена на удаленеи в 0.6.0)

---

### Q3: Надо ли фиксить найденные баги в рамках этой задачи?

**Контекст**: Найдено несколько потенциальных проблем:
- `Batcher.run()` без try/finally (блокировка при ошибке)  
- `deepEqual` не обрабатывает Date, RegExp, NaN, циклические ссылки
- `reduxDevtools` throws при отсутствии расширения
- `LocalState.DEFAULT_DRIVER = localStorage` — ошибка в SSR
- `useSignal` без `getServerSnapshot`

**Варианты:**
1. **Только тесты** — задокументировать баги, написать failing тесты, не фиксить
2. **Тесты + фиксы** — исправить найденные проблемы и покрыть тестами
3. **Тесты + фиксы критических** — исправить только блокирующие (Batcher try/finally) и покрыть всё тестами

**Нужно решение человека**: Выбор scope

Ответ: Тесты + фиксы критических

## Важные (влияют на Design)

### Q4: Нужна ли SSR-совместимость?

**Контекст**: Два места обращаются к browser-only API:
- `reduxDevtools.ts` → `window.__REDUX_DEVTOOLS_EXTENSION__`
- `LocalState.ts` → `localStorage`

Также `useSignal` не предоставляет `getServerSnapshot` для `useSyncExternalStore`.

**Вопрос**: Является ли SSR-совместимость требованием для этого релиза?

Ответ: Нет

---

### Q5: Какой уровень code coverage считать достаточным?

**Варианты:**
- **80%+** — стандартный минимум для open-source библиотек
- **90%+** — high-quality библиотека (рекомендуется для reactive primitives)
- **100% для публичного API, 80% для внутреннего** — компромисс

Ответ: 80

---

### Q6: Нужно ли тестировать внутренние API?

**Контекст**: Ряд компонентов экспортируются, но выглядят как внутренние (Batcher, ComputeCache, DependencyTracker, SyncObservable).

**Варианты:**
1. Тестировать только через публичный API (State, Computed, Effect, Signal)
2. Тестировать каждый компонент изолированно
3. Смешанный подход — отдельные unit-тесты для сложных внутренних компонентов (Batcher, ComputeCache), integration-тесты для остального

Ответ: Смешанный подход

### Q8: Что делать с `WriteableSignalLike` (опечатка)?

**Контекст**: В `signals.types.ts` интерфейс назван `WriteableSignalLike`, но стандартное написание — `WritableSignalLike`. Это breaking change для TypeScript пользователей, которые могут использовать этот тип.

Ответ: Исправить, пометить как deprecated

---

### Q9: Нужно ли выделить `Batcher` из зависимости `common → signals`?

**Контекст**: `reduxDevtools.ts` — единственный файл в `common`, который импортирует из `signals`. Это создаёт циклическую зависимость. Возможные решения:
1. Переместить `Batcher` в `common`
2. Переместить `reduxDevtools` в `signals`
3. Создать отдельный shared-модуль
4. Оставить как есть (работает на практике из-за lazy resolution)

Ответ: Да (выбери лучший способ)
